# Instructions for Claude Code

## Project Overview

Pi-hole AI Guardian: Real-time AI domain classifier + threat intel auto-sync daemon running on Raspberry Pi with Ollama.

**Tech Stack:** Python 3.11, sqlite3, requests, watchdog, Ollama (granite3.3:2b), systemd

**Key Locations:**
- Main code: `*.py` files in root
- Tests: `tests/test_*.py` (31 tests, all mocked, run with `make test`)
- Config: `config.py` (defaults), `config_local.py` (user config, generated at install)
- Systemd: `pihole-ai.service.tpl` (template, substituted at install)
- Build: `Makefile` (all build/daemon management targets)

## Build & Test

```bash
make install     # Create venv + install deps (pytest, requests, watchdog)
make test        # Run 31 tests (no external API calls, all mocked)
make clean       # Remove venv + caches
```

## Architecture

**Two daemon threads:**

1. **DomainWatcher** (`watcher.py`) — Tails `/var/log/pihole/pihole.log` in real-time
   - Extracts domain names from dnsmasq query lines
   - Checks StateDB for "seen before" (avoid re-classifying)
   - Sends to Ollama for classification
   - If malware/phishing/C2 + confidence > 80% → auto-block via PiholeClient
   - Logs all decisions to StateDB

2. **ThreatIntelSyncer** (`syncer.py`) — Runs on 6-hour interval
   - Fetches IOC feeds (Feodo, URLhaus, DigitalSide, OpenPhish)
   - Parses domains (handles hosts-file format, plain lists, JSON)
   - Deduplicates against StateDB (don't re-insert known threats)
   - Bulk-inserts new domains into Pi-hole gravity.db denylist
   - Logs sync history to StateDB

**Shared Resources:**
- `StateDB` (`state_db.py`) — SQLite: seen_domains, classifications, threat_domains, sync_log tables
- `PiholeClient` (`pihole_client.py`) — Reads FTL log, writes to gravity.db, triggers `pihole reloadlists`
- `OllamaClient` (`ollama_client.py`) — HTTP wrapper to Ollama /api/generate (with retry + timeout)
- `DomainClassifier` (`classifier.py`) — Builds structured prompt, sends to Ollama, parses JSON response

## Code Style

- **No comments** unless WHY is non-obvious (constraints, workarounds, subtle invariants)
- **Type hints** on function signatures
- **Logging** via stdlib logging (all modules have `logger = logging.getLogger(__name__)`)
- **Config imports** always start with: `try: from config_local import * except: pass; from config import *`
- **No error handling for impossible cases** — trust framework guarantees
- **DRY + YAGNI** — no premature abstractions, no feature flags for hypothetical features

## Configuration

**config.py** = static defaults (Ollama URL, thresholds, feed URLs, skip domains)
**config_local.py** = generated by setup wizard with user-specific values (paths, usernames, IPs, passwords)

All other code imports both and uses values transparently.

**Key Tunable Settings:**
- `BLOCK_CONFIDENCE_THRESHOLD` — only auto-block if confidence ≥ this (default 0.80)
- `CATEGORIES_TO_BLOCK` — which threat types to block (default: MALWARE, PHISHING, C2, RANSOMWARE)
- `THREAT_INTEL_INTERVAL_HOURS` — feed sync interval (default: 6)
- `SKIP_DOMAINS` — never classify these (pi.hole, localhost, google.com, etc)

## Testing Strategy

All tests use **mocking** — no external API calls:
- `test_state_db.py` — in-memory SQLite (tmpdir fixture)
- `test_pihole_client.py` — mock subprocess + SQLite
- `test_ollama_client.py` — mock requests library
- `test_classifier.py` — mock OllamaClient
- `test_threat_intel.py` — mock requests + parse fixtures

Fixtures:
- `tests/fixtures/sample_ftl.log` — fake FTL log lines for parser testing
- `tests/fixtures/sample_feed.txt` — fake threat feed (hosts-file format)

**Add new tests:** Follow TDD — write failing test first, then implement.

## Deployment

1. **On Pi-hole device:**
   ```bash
   git clone <repo> ~/pihole-ai && cd ~/pihole-ai
   make setup          # Interactive wizard
   ```

2. **Wizard configures:**
   - Username (for systemd service)
   - Pi-hole IP/hostname
   - Pi-hole password
   - Install directory
   - Ollama URL + model

3. **Wizard auto-installs:**
   - Venv + deps
   - Generates `config_local.py`
   - Generates `/etc/systemd/system/pihole-ai-$USER.service` from template
   - Creates `/etc/sudoers.d/pihole-ai-$USER` (allows user to run `pihole reloadlists` without password)
   - Starts systemd service

4. **Systemd service:**
   - Starts on boot
   - Restarts on crash (30s delay)
   - Runs as unprivileged user (from setup wizard)
   - Logs to journalctl + `logs/pihole-ai.log`

## Security Notes

- **No hardcoded secrets** — config_local.py with Pi-hole password is local-only, never committed
- **Ollama local-only** — all classification happens on-device
- **Pi-hole DB write** — only user running daemon can write (sudoers rule)
- **No external API calls** except threat intel feeds (public IOC lists)
- **SQLite locking** — thread-safe via connection pool per thread

## Common Debugging

### Daemon stuck or hung

```bash
make daemon-status
sudo journalctl -u pihole-ai-$USER -f
```

### Classification not working

```bash
# Check Ollama
curl http://localhost:11434/

# Check state DB
sqlite3 state.db "SELECT * FROM classifications ORDER BY classified_at DESC LIMIT 5"

# Check logs
make logs
```

### Threat intel sync failed

```bash
sqlite3 state.db "SELECT * FROM sync_log ORDER BY synced_at DESC LIMIT 5"
```

### Pi-hole denylist not reloading

```bash
# Test manually
sudo pihole reloadlists
echo $?  # Should be 0
```

## Important Constraints

**DO NOT:**
- Add cloud-based APIs or telemetry
- Modify threat intel feeds without testing
- Change SQLite schema without migration plan
- Block stdlib logging or requests library
- Hardcode any user-specific paths or IPs

**ALWAYS:**
- Run `make test` before committing
- Update config.py if adding new tunable setting
- Use StateDB for all persistence (no file-based state)
- Log decisions via logger (helps debugging)
